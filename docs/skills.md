# Skills 文档：AI 小说转剧本工具

本文档说明完成「AI 小说转剧本工具」所需要的核心能力，以及这些能力在项目中的具体体现。

---

## 1. Java 后端开发能力

### 为什么需要

本项目后端使用 Java 和 Spring Boot 实现，需要具备 Java 基础语法、面向对象设计、异常处理、集合操作和工程化开发能力。

### 在本项目中的体现

- 使用 Java 17 编写后端服务
- 使用 DTO 封装请求和响应
- 使用 Service 层承载业务逻辑
- 使用异常类处理业务错误
- 使用 Maven 管理依赖和构建

### 最低实现要求

- 能创建 Spring Boot 项目
- 能编写 Controller、Service、DTO
- 能处理基本异常
- 能通过 Maven 启动项目

### 进阶优化方向

- 增加单元测试
- 增加接口测试
- 优化对象建模
- 增加更完善的日志记录
- 引入更严格的代码规范

---

## 2. Spring Boot API 开发能力

### 为什么需要

项目需要提供 Web API 给前端调用，包括剧本生成、YAML 校验和健康检查接口。

### 在本项目中的体现

- `GET /api/health`
- `POST /api/scripts/generate`
- `POST /api/scripts/validate`
- 使用 JSON 请求和响应
- 使用全局异常处理统一错误返回

### 最低实现要求

- 能定义 REST API
- 能接收 JSON 请求体
- 能返回结构化 JSON 响应
- 能处理参数错误

### 进阶优化方向

- 增加 OpenAPI / Swagger 文档
- 增加请求日志
- 增加接口限流
- 增加统一响应编码
- 增加更细粒度错误码

---

## 3. AI API 调用能力

### 为什么需要

小说转剧本的核心能力依赖大模型完成，包括人物抽取、情节理解、场景拆分和剧本生成。

### 在本项目中的体现

- 通过 OpenAI-compatible Chat Completions API 调用大模型
- 使用环境变量配置 API Key、Base URL 和模型名称
- 支持没有 API Key 时的 mock fallback
- 解析 AI 返回内容并提取 YAML

### 最低实现要求

- 能发起 HTTP 请求
- 能设置 Authorization Header
- 能构造 messages 请求体
- 能读取模型返回文本
- 能处理 API 调用失败

### 进阶优化方向

- 支持流式输出
- 支持多模型切换
- 增加重试机制
- 增加超时控制
- 增加 token 预算管理

---

## 4. Prompt Engineering 能力

### 为什么需要

大模型输出质量高度依赖 prompt。项目要求输出可解析、符合 Schema 的 YAML 剧本，因此需要清晰、约束明确的提示词设计。

### 在本项目中的体现

- 生成剧本 prompt
- YAML 修复 prompt
- 要求 AI 只输出 YAML
- 要求 AI 按指定 Schema 输出
- 要求 AI 将小说叙事转换为剧本场景、动作和对白

### 最低实现要求

- 能写清楚任务目标
- 能给出输出格式约束
- 能明确禁止额外解释文字
- 能描述必要字段
- 能说明改编规则

### 进阶优化方向

- 分阶段 prompt
- 先生成 JSON 再转 YAML
- 增加自检 prompt
- 增加少样本示例
- 针对不同剧本类型设计不同 prompt

---

## 5. 长文本处理能力

### 为什么需要

用户输入至少 3 个章节的小说文本，文本可能较长，不能简单无限制地传给模型。

### 在本项目中的体现

- 按章节拆分小说
- 检查章节数量
- 对章节内容进行截断或摘要
- 将章节信息组织进 prompt

### 最低实现要求

- 能识别常见章节标题
- 能将小说拆分为章节列表
- 能判断章节数量是否满足要求
- 能控制传给模型的文本长度

### 进阶优化方向

- 分章摘要
- 分块处理
- Map-Reduce 式总结
- 引入向量检索
- 支持超长小说项目

---

## 6. YAML / Schema 设计能力

### 为什么需要

题目要求输出结构化剧本，并使用 YAML 格式。因此需要设计清晰、可编辑、可扩展的剧本 Schema。

### 在本项目中的体现

- 使用 `script` 作为根节点
- 定义 `metadata`
- 定义 `source`
- 定义 `characters`
- 定义 `scenes`
- 定义 `beats`
- 使用角色 ID 和场景 ID 进行引用

### 最低实现要求

- 能设计层级清晰的 YAML
- 能区分元信息、来源信息、人物和场景
- 能保证字段命名稳定
- 能让 YAML 便于人工阅读和编辑

### 进阶优化方向

- 提供 JSON Schema
- 支持多版本 Schema
- 支持 Schema 自动迁移
- 支持更多剧本类型
- 支持导出 Final Draft / Markdown

## 8. 错误处理与数据校验能力

### 为什么需要

AI 输出可能不稳定，用户输入也可能不规范。系统必须能够发现错误并给出可理解的反馈。

### 在本项目中的体现

- 少于 3 个章节时返回错误
- YAML 无法解析时返回错误
- 缺少关键字段时返回错误
- dialogue beat 缺少角色信息时返回错误
- AI 未配置时进入 mock 模式

### 最低实现要求

- 能校验用户输入
- 能捕获业务异常
- 能捕获系统异常
- 能返回统一错误格式
- 能展示错误信息给用户

### 进阶优化方向

- 自动修复 YAML
- 增加错误码
- 增加日志追踪 ID
- 增加 AI 输出质量评分
- 增加前端字段级提示

---

## 9. 产品拆解能力

### 为什么需要

两天内无法完成一个完整商业产品，因此需要将需求拆解为 MVP、P1 和 P2，优先完成核心闭环。

### 在本项目中的体现

- P0：输入小说并生成 YAML 剧本
- P1：自动修复、文档和测试
- P2：局部重写、历史记录、多格式导出

### 最低实现要求

- 能识别核心用户路径
- 能确定必须完成的功能
- 能砍掉非必要功能
- 能保证最终 Demo 可用

### 进阶优化方向

- 用户分层
- 场景化模板
- 商业化功能设计
- 多人协作编辑
- 版本管理

---

## 10. 两天交付能力

### 为什么需要

本项目有明确时间限制，需要在两天内完成可运行、可展示、可说明的版本。

### 在本项目中的体现

- 使用 Spring Boot 快速搭建后端
- 使用原生前端降低复杂度
- 不引入数据库
- 不做登录系统
- 使用 mock fallback 保证演示稳定
- 优先完成主链路

### 最低实现要求

- Day 1 完成项目骨架、章节解析、YAML 校验和 AI 调用
- Day 2 完成完整生成链路、前端交互和文档
- 最终项目可启动、可访问、可演示

### 进阶优化方向

- 增加自动化测试
- 增加 CI
- 增加 Docker 部署
- 增加线上演示环境
- 增加性能优化
